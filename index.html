SECTION 1Databricksとは（全体像）

Databricksは、企業のあらゆるデータを1か所に集約し、整え、分析し、機械学習・生成AIに活用するまでを単一プラットフォームで行える「データインテリジェンスプラットフォーム」です。

従来、企業のデータ活用には2つの別々の仕組みが必要でした。

  • データウェアハウス（DWH）：きれいに整えた構造化データを高速にSQL分析・BIする箱。信頼性は高いが、画像・テキストなど非構造化データやAI用途は苦手。
  • データレイク：あらゆる生データを安く大量にためる箱。AI/データサイエンス向きだが、品質保証がなくBIには信頼性が低い。

この2つを別々に持つと、データが二重管理になり（サイロ化）、コスト増・鮮度低下・ガバナンスの分断が起きます。Databricksはこの両者の良いとこ取りをした「レイクハウス」という1つの箱に統合することを提案しました。これがDatabricksを理解する最重要キーワードです。

💡 ここだけ押さえる

Databricks ＝ 「レイクハウス」という1つの基盤の上で、データ整備（エンジニアリング）・分析（BI）・AI（機械学習/生成AI）を同じデータ・同じガバナンスで回せること。「データとAIを分断させない」のが価値の核心です。

SECTION 2歴史と会社背景

Databricksの強みの源泉は、その出自にあります。「分散処理エンジンApache Sparkを生んだ研究者たちが、その商用版として創業した会社」という点が、技術的な信頼性とAI志向の背景になっています。

起源：UC Berkeley と Apache Spark

2009年ごろ、カリフォルニア大学バークレー校の研究室「AMPLab」で、Matei Zaharia氏らが大規模データを高速に分散処理するエンジン Apache Spark を開発しました（2010年にオープンソース化）。MapReduceの非効率を解決するためにメモリを活用する設計が画期的で、ビッグデータ処理の事実上の標準になっていきます。

2013年、このSparkの中心メンバー7名が、Sparkを企業が使いやすい形で提供するために Databricks を創業しました。

SECTION 3アーキテクチャとコア概念

勉強会で頻出する用語の中核を、初学者向けに整理します。レイクハウス＝家全体、Delta Lake＝土台、Unity Catalog＝管理人、Photon＝エンジンとイメージすると掴みやすいです。

3-1. レイクハウス（Lakehouse）

DWHとデータレイクの利点を1つに統合したアーキテクチャ。安価なクラウドストレージにオープン形式でためつつ、DWH並みの信頼性・高速SQL・ガバナンスを実現します。「データの単一の正（Single Source of Truth）」を作ることが狙いです。

観点データウェアハウスデータレイクレイクハウス
得意BI・SQL分析生データ蓄積・AI両方
データ種別構造化中心何でも何でも
信頼性（ACID）高い低い高い
コスト高め安い安いストレージ＋必要時計算
ロックインされやすい少ない少ない（オープン形式）

出典：What is a data lakehouse?（公式ドキュメント）

3-2. Delta Lake（信頼性の土台）

レイクハウスのストレージの土台となるオープンな技術。安価なファイル（Parquet）にメタデータ層を足すことで、データベース並みの信頼性をデータレイクにもたらします。

  • ACIDトランザクション：複数人が同時に読み書きしてもデータが壊れない（中途半端な状態が見えない）。
  • タイムトラベル：「3日前のテーブル状態」をバージョンや時刻で遡って参照できる。監査・障害復旧に有効。
  • スキーマ管理：列の型を強制（enforcement）しつつ、必要に応じて拡張（evolution）できる。
📌 Delta と Iceberg（よく聞かれる論点）

業界にはもう一つの主要オープン形式 Apache Iceberg があります。Databricksは UniForm（現：Iceberg reads） という仕組みで、Delta LakeのテーブルをIcebergクライアントからも読めるようにしています（※外部Icebergからは読み取り専用という制約あり）。Tabular買収もあり、Databricksは「形式の壁を越える相互運用」を進めています。

出典：Read Delta Lake tables with Iceberg clients（公式）

3-3. Unity Catalog（統一ガバナンス＝管理人）

データとAI資産を横断的に一元管理する仕組み。「誰が何にアクセスできるか」「このデータはどこから来てどこで使われているか」を統一的に扱います。データ活用を組織で安全に広げるうえで極めて重要です。

  • 3階層の名前空間：カタログ.スキーマ.テーブル でデータを整理。
  • アクセス制御：使い慣れたSQL構文で、行フィルタリングや列マスキング（個人情報の隠蔽）まで設定可能。
  • データリネージ：どのジョブ・ノートブックがデータを作り、どのダッシュボードが使っているかを列レベルまで自動追跡。
  • 監査ログ：誰がいつ何にアクセスしたかを記録。

出典：What is Unity Catalog?（公式）

3-4. Photon / Databricks SQL（高速エンジンとDWH機能）

Photonは、DatabricksがC++でゼロから作り直した高速クエリエンジン。データを列指向のまとまりで処理し、SQLやETLを大幅に高速化します（公式は他クラウドDWH比で高い価格性能を主張）。既存のSQL/Python/Scalaコードを書き換えずに使えるのが利点です。

Databricks SQLは、レイクハウス上でDWHのようにSQL分析・BIを行う機能群。その計算基盤が SQLウェアハウスで、性能と運用負荷に応じて Serverless / Pro / Classic の3タイプから選べます（Serverlessは起動が速く運用をDatabricksに任せられる）。

出典：What is Photon?（公式）／Data warehousing on Databricks（公式）

3-5. コンピュートと「2つのプレーン」（重要な構造）

Databricksは コントロールプレーン と コンピュートプレーン という2層で動きます。セキュリティやコストを理解するうえで重要な概念です。

  • コントロールプレーン：画面・ジョブ管理・設定などDatabricksが運用するバックエンド。
  • コンピュートプレーン（計算が走る場所）：
    • Classic：計算リソースが自社のクラウドアカウント内で動く（ネットワークを自社で分離できる）。
    • Serverless：計算リソースがDatabricks側で動く（起動が速く、容量管理やパッチをお任せできる）。
📌 用語メモ

旧称「データプレーン（Data Plane）」が現在の「コンピュートプレーン（Compute Plane）」に相当します（現行ドキュメントはCompute Planeで統一）。勉強会でどちらの語が出ても同じものを指していると考えてOKです。

出典：High-level architecture（公式）

3-7. マルチクラウド対応

Databricksは AWS・Microsoft Azure・Google Cloud のどこでも同じように動きます。プラットフォーム自体に本質的な差はなく、違いは各クラウドとの統合・課金方法にあります。

  • Azure Databricks：Microsoftの第一級（ファーストパーティ）サービス。Azureの請求に統合され、Microsoftエコシステムとの結びつきが強い。
  • AWS / Google Cloud：サードパーティ統合として提供。それぞれS3／BigQuery等のネイティブサービスと相互運用できる。

出典：Cloud Partners（公式）／Azure版は Microsoft Learn に一次情報あり

3-8. オープンソース戦略

Databricksの特徴は主要技術を積極的にオープンソース化している点です。「ベンダーロックインを避けたい」という顧客の要望に応える戦略で、競合との差別化にもなっています。

  • Apache Spark（分散処理）／Delta Lake（ストレージ）／MLflow（ML管理）はいずれもOSS。
  • Unity Catalog も2024年6月にApache 2.0でOSS化（Linux Foundation傘下でホスト）。
  • データは標準形式（Delta/Parquet/Iceberg）で保存されるため、他のツールからも読める。

出典：Open Sourcing Unity Catalog（公式ブログ）

SECTION 4データ活用（データ基盤・ETL）

データを「集めて・整えて・流し続ける」ためのDatabricksの機能群です。2024年以降、これらは Lakeflow というブランドに統合されました。

取り込み

Lakeflow Connect

SQL Server・Salesforce・Workday等から、ノーコード／CDCでデータを取り込む。

取り込み

Auto Loader

クラウドストレージに届いたファイルを自動で検知し、増分だけ取り込む。大規模処理に強い。

変換

Lakeflow Pipelines

旧 Delta Live Tables（DLT）。宣言的にパイプラインを書くと、依存関係・スケーリング・品質チェックを自動化。

実行管理

Lakeflow Jobs

旧 Workflows。ノートブックやパイプラインをDAGで組み、スケジュール実行・監視する。

📌 名称メモ

勉強会で 「DLT」＝Lakeflow Pipelines、「Workflows」＝Lakeflow Jobs という旧称が出ることがあります。中身は連続しているので混乱しないように。

出典：Introducing Lakeflow（公式ブログ）／Auto Loader（公式）

SECTION 5AI活用(機械学習・生成AI)

「データの隣でAIを動かせる」のがDatabricks最大の差別化ポイント。データを移動せず、ガバナンスを効かせたままAI/MLを開発・運用できます。

機械学習の土台：MLflow

Databricks発のOSSで、機械学習の実験管理・モデル管理・MLOpsのデファクト標準。「どのデータ・パラメータでどの精度が出たか」を記録し、モデルを本番にデプロイ・監視するまでを一貫管理します。

生成AIの中核：Mosaic AI

MosaicML買収を土台にした生成AI機能群。自社データを使ったAIアプリを作る道具がそろっています。

Model Serving

自社モデルや基盤モデルをAPIとして提供。

Vector Search

RAGの中核。社内文書をベクトル化し、Deltaと自動同期して常に最新を検索。

Agent Framework

RAGアプリやAIエージェントの構築・評価・運用（LLMOps）。

AI Gateway

LLM利用の統制。使用量・レート制限・PII検出・コスト把握。

誰でも使えるAI：AI/BI

  • Genie：自然言語（日本語の質問など）でデータに問い合わせ、その場で分析・可視化できる。非エンジニアの活用を広げる。
  • Databricks Assistant：コード生成や説明を助けるAIコパイロット。

自社LLM：DBRX

2024年3月公開の自社製オープンLLM。Mixture-of-Experts（MoE）方式で、総パラメータ1,320億のうちトークンあたり360億だけを使う効率的な設計。「オープンで高性能なモデルを提供できる会社」という技術力の象徴です。

💡 生成AI活用の典型像（RAG）

「社内データ（Delta Lake）→ Vector Searchでインデックス化 → 質問に関連する情報を取得 → LLMに渡して回答生成 → AI Gateway/Unity Catalogで統制」という流れを、すべて同じ基盤・同じガバナンスの中で組めるのがDatabricksの売りです。

出典：Mosaic AI generative AI capabilities（公式）／Announcing DBRX（公式ブログ）

SECTION 7弱み・課題・注意点

導入を検討するうえで現実的に知っておくべき「つまずきどころ」です（第三者レビューを含む客観的評価）。

⚠️ よく指摘される弱み

  • 学習コストが高い：機能が多く、エンジニア前提の作り。初学者には急な学習曲線。
  • コスト管理が難しい：従量課金＋自動スケールで、油断すると費用が膨らみやすい。予測が難しい。
  • 初期セットアップが複雑：クラウド連携・ネットワーク・権限設計など考慮点が多い。
  • 非エンジニアには敷居が高い：ノーコードのBI専用ツールに比べると、一般ユーザーが使いこなしにくい面。
  • クラウド/リージョン差：新機能の提供時期や対応がクラウド・地域で異なる場合がある。
  • 専門人材が必要：SQL＋Python/Scala＋ML/データ基盤の知識を持つ人材・体制が要る。

✅ 弱みへの向き合い方

  • Serverlessや既定のテンプレートでセットアップを簡素化できる。
  • 予算アラート・タグ付け・ジョブ用コンピュートの活用でコストのガードレールを敷ける。
  • Genie/AssistantなどAI支援で非エンジニアの利用を補える。
  • 小さく始めて（PoC）段階的に拡大するのが定石。
  • 公式トレーニング・認定資格でスキルを計画的に獲得。
⚠️ 一番のリアル課題は「コスト」

第三者レビューで最も繰り返し挙がるのがコストの読みにくさです。勉強会では「コストを予測・最適化する具体策」「サーバーレスとクラシックの使い分け」を必ず聞いておくと実務で効きます（→ 11章 に質問例）。

SECTION 8競合比較と立ち位置

「Databricks vs ○○」は勉強会で必ず話題になります。重要なのは「どれか一つを選ぶ」ではなく「ワークロードごとに使い分け・併用する」のが2026年の主流という点です。

8-1. Databricks vs Snowflake（最大の比較対象）

観点DatabricksSnowflake
出自・設計レイクハウス（Spark由来）。データは自社ストレージにオープン形式クラウドDWH。Snowflake管理環境内（近年Icebergで外部形式も）
得意領域ML/AI・大規模ETL・ストリーミング・非構造化SQL分析・BI・高同時実行
使いやすさコード中心。エンジニア向けで学習曲線が急SQL中心。アナリストに優しくプラグ＆プレイ
コスト傾向大規模ETLで有利になりやすいSQL分析でシンプル・運用が軽い
ロックイン少なめ（オープン形式）されやすい（近年改善）
📌 一行で言うと

「AI・大規模データ処理ならDatabricks、SQL分析・BIの手軽さならSnowflake」。両者は機能を相互に拡張中で境界は曖昧になりつつあります。

8-2. AWSとの関係 ―「パートナーでもあり競合でもある」

DatabricksはAWS上で動く（＝S3・IAM・VPCと統合するパートナー）一方で、AWS自身のネイティブサービスとは機能が重なります。

役割AWSネイティブDatabricksの立ち位置
DWH/SQL分析Redshift / AthenaDatabricks SQLが対抗。ただしSQL専用ならRedshift/Athenaが手軽な面も
分散処理(Spark)EMRSpark最適化・UI・ガバナンスでDatabricksが優位とされる
ETL/カタログGlueLakeflow＋Unity Catalogが対抗（Glue連携も可能）
機械学習SageMakerMLflow＋Mosaic AIが対抗

ポイントは「単一のAWSサービスでDatabricks全体を置き換えられるものはない」こと。AWS派は EMR＋Glue＋SageMaker＋Lake Formation の組み合わせで近いことができますが、統合度・運用の手間で差が出ます。

8-3. Google Cloudとの関係

同様に、GCP上で動く一方、BigQuery（サーバーレスDWH）や Vertex AI（ML/生成AI） と競合します。実務では「Databricks＝データエンジニアリング/ETL/レイクハウス」「BigQuery/Vertex＝サーバーレスSQL分析・基盤モデル」と役割分担で併用する企業が多い、という整理がよくされます。

8-4. Azure / Microsoft Fabric との関係

前述のとおり Azure Databricksはマイクロソフトの第一級サービスで、両社は戦略提携を継続。Microsoftの統合分析基盤 Fabric とは「直接対決」というより役割分担（Fabric＝全社向けの手軽な分析/BI、Databricks＝大規模エンジニアリング・カスタムML）。Unity CatalogとOneLakeの連携など、共存の動きも進んでいます。

8-5. 結局どう選ぶ？（向き／不向き）

✅ Databricksが向くケース

  • 大規模ETL・複雑なパイプライン・ストリーミング
  • ML/AI・生成AIが主目的(データの隣でAIを動かしたい)
  • データとAIの統一ガバナンスが必要
  • オープン形式重視・マルチクラウド・ロックイン回避
  • コードファーストなエンジニアチームがいる

⚠️ 他が手軽なケース

  • 目的がSQL分析・BIダッシュボード中心 → Snowflake / BigQueryが手軽
  • 専任のデータエンジニアが乏しい → 学習コストが重荷に
  • すでに特定クラウドのネイティブ構成で完結している

出典：Flexera: Snowflake vs Databricks／3製品比較ガイド（2026）／各クラウド公式ブログ（中立性のため複数ソースを参照）

SECTION 9料金の基礎（DBU）

Databricksのコストは「分かりにくい」と言われがちですが、二層構造と DBU という考え方を押さえれば見通せます。

💰 コストの基本式

総コスト ＝ ① クラウドのインフラ費用（VM・ストレージ等：クラウド事業者へ）＋ ② Databricks利用料（消費したDBU × DBU単価：Databricksへ）

DBU（Databricks Unit）とは

処理能力を正規化した従量課金の単位。インスタンス種別やクラウドが違っても一貫して計量でき、秒単位で消費されます。「処理パワーをどれだけ使ったか」のメーターだとイメージしてください。

ワークロードで単価が変わる

ワークロード種別用途単価の傾向
Jobs Compute自動化・スケジュール実行のETL最も安い
All-Purpose Compute対話的なノートブック開発Jobsの2〜3倍と高め
SQL（SQL Warehouse）SQL分析・BI（Serverless/Pro/Classic）タイプで差
Pipelines（DLT）等宣言的パイプライン専用単価
💡 コスト最適化の勘所
  • 本番の定期処理は安いJobs Computeで。開発の高いAll-Purposeを動かしっぱなしにしない（自動終了を設定）。
  • プラン階層（Standard/Premium/Enterprise）でも単価が変わる。
  • 具体的な単価は変動が激しいので、必ず公式の料金ページで最新を確認。

出典：Databricks Pricing（公式）／CloudZero: Databricks Pricing（単価の参考値は第三者まとめ・要変動注意）

SECTION 1目玉：LTAP / Lakebase（運用×分析の統合）

LTAP（Lake Transactional/Analytical Processing） Coming Soon 発表

トランザクション処理（OLTP）と分析（OLAP）、さらにストリーミングを「単一のデータコピー」（オブジェクトストレージ上のDelta/Iceberg）で扱う新アーキテクチャ。これにより、従来必要だったETLパイプライン・レプリカ・CDCを不要にするという構想。Databricksは「世界初のLTAP」と表現し、「パイプラインの終焉」を掲げました。

ガバナンスはUnity Catalogで統一。提供は Lakebaseの一部として「coming soon」（＝発表・プレビュー段階で、まだGAではない）。

出典：Databricks プレスリリース（LTAP, 6/16）／SiliconANGLE

Lakebase（サーバーレスPostgres）の新機能 本体GA 新機能は発表

2025年のNeon買収由来のサーバーレスPostgres。本体は2026年初頭にGA済みで、「数千社が利用、1日1,200万回のDB起動」を公表。今回の新機能として、クロスクラウド/クロスリージョンの災害復旧（DR）、Gitスタイルのブランチ＆スナップショット、自律的なDB運用を追加。AIエージェントの「記憶層」としても位置づけられます。

出典：同 プレスリリース（6/16）

Lakehouse//RT ＋ 新エンジン「Reyden」 Beta ステータス要確認

リアルタイム分析エンジン。小規模で約10ms、大規模でも100ms未満のレイテンシ、数万規模の同時ユーザー/エージェントに対応し、既存ウェアハウスのドロップイン置換を狙うとされます。基盤の新実行エンジン「Reyden」（共同創業者 Reynold Xin に由来する命名）は既存比 最大16倍の性能を主張。

注：この項目は主にテックメディア（SiliconANGLE）報道に基づき、公式単独ソースでの裏取りが弱め。ステータス（beta/launched）の表記にも揺れがあるため要確認です。

出典：SiliconANGLE（6/16）

SECTION 3Genie（AI/BIエージェント）

Genie One 発表

実業務データに根ざして作業を自動化する「エージェント型の同僚（co-worker）」。Ali Ghodsi CEO は「Genie One は“計算する”、他の多くのエージェントは“復唱する”だけ」と差別化を強調しました。

Genie Ontology 発表

社内外データから継続的に学習する「ライブのコンテキスト層」。企業内の情報探索における“PageRank”のような役割と説明されています。

Genie App Builder ／ Lakeflow Designer 発表 ステータス要確認

Genie App Builder：自社のライブデータで動く、統制済みのローコード・アプリ構築環境。Lakeflow Designer：自然言語でデータパイプラインを設計・自動化。いずれも正式なGA/Previewの別は明示が弱く、現時点では発表ベースです。

出典：SiliconANGLE／Tech Times（Day2まとめ）

モデル・ツール・エージェント・MCPサービス・外部AI資産を一元統制するガバナンス層。「誰がアクセスできるか」だけでなく「AIが何を実行できるか」を管理します。統一トレーシング・予算管理・ハードな支出上限・スマートルーティング、監視基盤 Lakewatch との連携を提供。
